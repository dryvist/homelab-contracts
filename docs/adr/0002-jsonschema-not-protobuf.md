# 0002 — JSON Schema, not Protobuf or CUE

- Status: accepted
- Date: 2026-05-10

## Context

The contract format had to satisfy:

1. Generated by OpenTofu (HCL → JSON via `local_file` resource)
2. Consumed by Ansible (`ansible.utils.validate` engine `jsonschema`)
3. Human-readable, since this is a small homelab not a billion-row event stream
4. CI-validatable with off-the-shelf tooling

Candidates considered: JSON Schema, Protobuf, CUE, Avro, OpenAPI.

## Decision

JSON Schema (draft 2020-12).

| Format | Why not |
| --- | --- |
| Protobuf | Codegen step on every consumer; binary wire format wrong for an inventory artifact; no native HCL support |
| CUE | Beautiful, but adds a tool the team doesn't already run; less Ansible ecosystem support |
| Avro | Same as Protobuf; designed for streaming, not a single artifact |
| OpenAPI | Wrong semantic — describes APIs, not data shapes |
| JSON Schema | OpenTofu writes JSON natively; Ansible has first-class validation; `check-jsonschema` runs in CI in seconds |

Draft 2020-12 specifically because it gives us `$defs` references (cleaner than `definitions`)
and fixes long-standing `additionalProperties` semantics around `$ref`.

## Consequences

- Positive: one human-readable file; consumers need no codegen.
- Positive: `check-jsonschema` is a single CLI; no language runtime required in CI.
- Negative: JSON Schema's expressiveness has limits — we can't enforce e.g.
  "every `pbs_targets[*].sync_targets[*]` references a key that exists in `pbs_targets`."
  Cross-reference invariants are checked by the consumers' own playbook logic.
